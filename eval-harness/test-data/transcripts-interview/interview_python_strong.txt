[Candidate]: Sure. At my last role I owned a payments reconciliation service in Python using FastAPI. [Interviewer]: How did you handle scale and failures?

[Candidate]: We did about 2 million events a day. I added a dead-letter queue for poison messages, exponential backoff on retries, and structured logging with correlation IDs so we could trace a single transaction across services. For scale, the consumers were stateless and ran in containers on ECS, so we just scaled horizontally. The Postgres write path was the bottleneck, so I batched inserts and added a read replica for the reporting queries.

I'd look at whether one service processed a duplicate or missed an event — usually it's an ordering or idempotency issue. I'd reconcile against the source-of-truth ledger and add a regression test capturing that exact event sequence so it can't recur.
